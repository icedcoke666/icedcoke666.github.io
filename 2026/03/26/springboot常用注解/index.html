@configuration

  • 作用：标记一个类为 Spring 配置类，用来用 Java 代码代替 XML，在里面写  @Bean  方法，让 Spring 帮你创建、管理对象（Bean）。
  1. 核心作用（大白话）
  • 告诉 Spring：这个类是配置类，里面有 Bean 定义
  • 等价于以前的  applicationContext.xml 
  • 内部用  @Bean  方法，返回的对象会被 Spring 管理（单例、注入等）
  1. 简单示例
  1. 关键特性（你必须懂）

1. 本质是特殊的 @Component

  • 内部自带  @Component ，会被 Spring 扫描到
    2. CGLIB 代理（保证单例）
  • 配置类会被代理，内部多次调用  @Bean  方法，永远返回同一个单例
    3. 配合 @Bean 一起用
  • 没有  @Configuration ， @Bean  通常不生效
    4. Spring Boot 里的用法
  • 启动类上的  @SpringBootApplication  已经包含  @Configuration 
  • 自己写配置类直接加  @Configuration  即可
  1. 和 @Component 的区别（面试常问）
  • @Configuration：配置类，用来集中定义 Bean、依赖、整合第三方库
  • @Component / @Service / @Controller：业务组件，自己的类交给 Spring 管理
  1. 常用场景
  • 配置 Redis、MyBatis-Plus、线程池、MQ 等第三方 Bean
  • 整合第三方库（Druid、Elasticsearch 等）
  • 做 AOP、拦截器、全局配置

@SpringBootApplication

  • 作用：SpringBoot 启动类，标记这个类是 SpringBoot 应用的入口。
  1. 包含：@configuration、@EnableAutoConfiguration、@ComponentScan
  • @Configuration：标记为配置类
  • @EnableAutoConfiguration：开启自动配置
  • @ComponentScan：开启组件扫描，默认扫描当前包及其子包下的所有类

@ConponentScan

  • 作用：开启组件扫描，默认扫描当前包及其子包下的所有类。

 @ComponentScan  到底是干什么的？

一句话：
告诉 Spring：去哪些包下面，扫描带有 @Component、@Service、@Controller、@Repository 这些注解的类，把它们变成 Bean 放进容器里。

  1. 核心作用
  • 你写了  @Service 、 @Controller ，Spring 怎么知道要去管它们？
  • 靠的就是  @ComponentScan  指定扫描路径。
  • 不扫，Spring 根本看不见这些类，也就不会创建 Bean。
  1. 默认规则（非常重要）

如果你什么都不写，比如：

它内部已经自带  @ComponentScan ，并且默认扫描：

当前启动类所在的包，以及它下面所有子包

所以平时你不用自己写，也能正常用。

  1. 什么时候需要自己写？

只有一种常见情况：
你把类放到了启动类包的外面

比如：
plaintext
com.xxx.demo ← 启动类在这里
com.xxx.common ← 不在子包下面  

这时 Spring 扫不到  com.xxx.common  里的类，你就要手动写：

  1. 简单记（背这句就够）
  •  @ComponentScan  = Spring 的“搜索范围”
  • 不指定，就扫启动类所在包及子包
  • 指定了，就扫你写的那些包
  • 扫到的  @Component  系列类，都会变成 Bean 被管理

做项目一定会遇到的场景讲，什么时候要把类放到启动类包外面？

  1. 最常见、90% 项目都这么干

一定会做这种结构：

plaintext

com.xxx
├── common ← 公共模块（工具类、通用配置、通用异常）
├── system ← 系统管理模块
├── order ← 订单模块
└── gateway ← 网关模块  

每个模块都是独立的子工程（Maven 多模块）。

启动类在：

plaintext

com.xxx.gateway
└── GatewayApplication.java ← 启动类
 
但你要用到  common  包里的工具类、通用 Bean，
而  common  包 不在 gateway 包下面，

  1. 还有几种典型情况

1. 写了一个公共 SDK / 工具包给别人用

  • 别人的启动类在他们自己的包下
  • 你的类不在他们包里面
    → 别人必须用  @ComponentScan  扫你的包
    2. 项目拆分成微服务
  • 服务 A 的启动类在  com.xxx.a 
  • 公共 API、实体类在  com.xxx.api 
    → 服务 A 要扫  com.xxx.api 
    3. 老项目迁移、包结构不规范
  • 以前的代码包名很乱
  • 不在启动类下面
    → 只能手动扩大扫描范围
  1. 一句能背的总结

当我们使用 多模块项目、微服务、公共依赖包 时，
通用类、工具类、公共配置所在的包，
不在启动类所在包及其子包下，
这时候就必须用  @ComponentScan  手动指定扫描这些外部包。

  1. 怎么理解最简单？

就记一句话：
Spring 只认自己家门口的，别人家的东西，要你主动请它进来扫。
