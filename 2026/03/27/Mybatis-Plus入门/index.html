今日知识点：Mybatis-Plus入门

快速入门

单表的CRUD功能代码重复度很高，也没有什么难度。而这部分代码量往往比较大，开发起来比较费时。
因此，目前企业中都会使用一些组件来简化或省略单表的CRUD开发工作。目前在国内使用较多的一个组件就是MybatisPlus.
官方网址：MybatisPlus

2.mybatis-plus基本注解

  1. @TableName
  • 作用：指定实体类对应的数据库表名
  • 场景：类名和表名不一致时必须加
  1. @TableId
  • 作用：标记主键字段
  • 常用属性：
  • type = IdType.AUTO ：数据库自增
  • type = IdType.INPUT ：手动输入
  • type = IdType.ASSIGN_ID ：雪花算法（MP 默认）
  • type = IdType.ASSIGN_UUID ：UUID
  1. @TableField
  • 作用：指定字段与列的映射关系
  • 常用场景：
  1. 字段名与列名不一致
  1. 该属性不是数据库字段，再数据库中不存在，MP不参与SQL拼接（最常用）
  1. 开启乐观锁
  1. 自动填充（创建时间、更新时间）
  1. @TableLogic 逻辑删除注解
  • 作用：标记逻辑删除字段
  • 执行 removeById 会变成 UPDATE 而非 DELETE
  1. 原生兼容，MP可用Mybatis的所有注解

@Select / @Insert / @Update / @Delete

  • 作用：注解方式写 SQL

@Param

  • 作用：给参数命名，多参数时必须

极简速记版

  • @TableName → 对应表名
  • @TableId → 主键
  • @TableField → 字段映射 / 非数据库字段 / 自动填充
  • @TableLogic → 逻辑删除
  • @Mapper → Mapper 接口
  • @MapperScan → 扫描 Mapper
  • @Param → 参数命名

3.乐观锁介绍以及MP的实现原理

  1. 什么是乐观锁？

一句话：
乐观锁就是：假设别人不会同时改这条数据，只有真的冲突时才不让更新。

生活例子（秒杀/并发修改）

你和朋友同时改同一个用户余额：

  1. 你查到余额：100
  2. 朋友也查到余额：100
  3. 你扣 20 → 想改成 80
  4. 朋友扣 30 → 想改成 70
    如果没有锁，最后结果可能是 80 或 70，数据错乱。

乐观锁的解决方式：
给这条数据加一个 version 版本号

  • 初始 version = 1
  • 你更新时，必须带上 version=1
  • 更新成功后，version 自动变成 2
  • 朋友再想拿 version=1 去更新，就会失败

只有一个人能修改成功，这就是乐观锁。

  1. 乐观锁 vs 悲观锁（简单记）
  • 乐观锁：用 version，不加锁，性能高，适合读多写少
  • 悲观锁：用 for update，锁住整行，别人不能改，性能低

日常项目高并发场景：基本都用乐观锁

11.mybatis-plus的Db静态工具与IService接口方法区别

一句话总结：Db是静态工具，不用注入、不用写Service，适合快速/跨表/避免循环依赖；IService是Service层接口，需注入、支持事务/扩展，适合标准业务开发。

一、核心定位

  • Db（静态工具类）
  • 包： com.baomidou.mybatisplus.extension.toolkit.Db （3.4.0+）
  • 定位：无注入、无Service、静态调用的轻量CRUD工具
  • 本质：基于反射+SqlSession的全局静态门面，线程安全
  • IService（Service层接口）
  • 包： com.baomidou.mybatisplus.extension.service.IService 
  • 定位：标准业务层接口，配合 ServiceImpl 使用
  • 本质：基于 BaseMapper 封装，支持事务、批量、扩展

二、关键区别（一眼看懂）

  1. 使用方式（最大不同）
  • Db：静态调用，无需注入、无需自定义Service/ServiceImpl
    java

// 直接用，不用@Autowired
User user = Db.getById(1L, User.class);
List list = Db.lambdaQuery(User.class).like(…).list();  

  • IService：注入使用，必须定义 XxxService extends IService  +  XxxServiceImpl extends ServiceImpl<XxxMapper, T> 
    java

@Autowired
private UserService userService;
User user = userService.getById(1L);  

  1. 参数差异
  • Db：查询/删除必须传Class（反射拿表信息）；save/update传对象即可（从对象取类型）
    java

// ✅ 必须传User.class
Db.getById(1L, User.class);
Db.removeById(1L, User.class);
// ✅ 传对象，不用Class
Db.save(new User());
Db.updateById(new User());  

  • IService：泛型已绑定实体，方法无需传Class，直接用
    java

userService.getById(1L);
userService.removeById(1L);  

  1. 事务与扩展
  • Db：默认无事务；不支持自定义业务方法、AOP、事务注解
  • IService：天然支持事务（ @Transactional ）；可在Service里加业务逻辑、扩展方法、拦截器
  1. 适用场景
  • Db 优先用
  • 快速原型、脚本、工具类、测试
  • 跨表查询、避免Service循环依赖
  • 极简单单表操作，不想写Service
  • 非核心、非事务场景
  • IService 优先用
  • 标准业务开发、Controller→Service→Mapper分层
  • 事务、批量、分页、复杂业务逻辑
  • 需扩展、复用、AOP、权限控制
  • 团队协作、规范项目
  1. 功能覆盖
  • 两者CRUD、条件构造器、分页、链式调用基本一致
  • IService 更偏向业务语义（save/remove/get）；Db 更贴近数据操作

三、一句话选型

  • 图省事、跨表、防循环依赖 → 用Db
  • 标准业务、要事务、要扩展 → 用IService
