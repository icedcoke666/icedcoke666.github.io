今日知识点：Mybatis-Plus入门

快速入门

单表的CRUD功能代码重复度很高，也没有什么难度。而这部分代码量往往比较大，开发起来比较费时。
因此，目前企业中都会使用一些组件来简化或省略单表的CRUD开发工作。目前在国内使用较多的一个组件就是MybatisPlus.
官方网址：MybatisPlus

2.mybatis-plus基本注解

  1. @TableName
  • 作用：指定实体类对应的数据库表名
  • 场景：类名和表名不一致时必须加
  1. @TableId
  • 作用：标记主键字段
  • 常用属性：
  • type = IdType.AUTO ：数据库自增
  • type = IdType.INPUT ：手动输入
  • type = IdType.ASSIGN_ID ：雪花算法（MP 默认）
  • type = IdType.ASSIGN_UUID ：UUID
  1. @TableField
  • 作用：指定字段与列的映射关系
  • 常用场景：
  1. 字段名与列名不一致
  1. 该属性不是数据库字段，再数据库中不存在，MP不参与SQL拼接（最常用）
  1. 开启乐观锁
  1. 自动填充（创建时间、更新时间）
  1. @TableLogic 逻辑删除注解
  • 作用：标记逻辑删除字段
  • 执行 removeById 会变成 UPDATE 而非 DELETE
  1. 原生兼容，MP可用Mybatis的所有注解

@Select / @Insert / @Update / @Delete

  • 作用：注解方式写 SQL

@Param

  • 作用：给参数命名，多参数时必须

极简速记版

  • @TableName → 对应表名
  • @TableId → 主键
  • @TableField → 字段映射 / 非数据库字段 / 自动填充
  • @TableLogic → 逻辑删除
  • @Mapper → Mapper 接口
  • @MapperScan → 扫描 Mapper
  • @Param → 参数命名

3.乐观锁介绍以及MP的实现原理

  1. 什么是乐观锁？

一句话：
乐观锁就是：假设别人不会同时改这条数据，只有真的冲突时才不让更新。

生活例子（秒杀/并发修改）

你和朋友同时改同一个用户余额：

  1. 你查到余额：100
  2. 朋友也查到余额：100
  3. 你扣 20 → 想改成 80
  4. 朋友扣 30 → 想改成 70
    如果没有锁，最后结果可能是 80 或 70，数据错乱。

乐观锁的解决方式：
给这条数据加一个 version 版本号

  • 初始 version = 1
  • 你更新时，必须带上 version=1
  • 更新成功后，version 自动变成 2
  • 朋友再想拿 version=1 去更新，就会失败

只有一个人能修改成功，这就是乐观锁。

  1. 乐观锁 vs 悲观锁（简单记）
  • 乐观锁：用 version，不加锁，性能高，适合读多写少
  • 悲观锁：用 for update，锁住整行，别人不能改，性能低

日常项目高并发场景：基本都用乐观锁

可以看到默认值是classpath*:/mapper/**/*.xml，也就是说我们只要把mapper.xml文件放置这个目录下就一定会被加载。

  • 作用：指定 MyBatis 加载自定义 SQL 的 Mapper XML 文件路径。
  • 路径含义：
  • classpath*: ：扫描所有类路径下的资源（包括 JAR 包）。
  • /mapper/**/*.xml ：匹配 resources/mapper 目录下所有子目录的 .xml 文件。
  • 文件存放：
  • 如图所示， UserMapper.xml 需放在 src/main/resources/mapper/ 目录下，才能被自动加载。

无论是QueryWrapper还是UpdateWrapper在构造条件的时候都需要写死字段名称，会出现字符串魔法值。这在编程规范中显然是不推荐的。​
那怎么样才能不写字段名，又能知道字段名呢？​
​
